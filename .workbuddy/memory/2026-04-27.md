## 任务1：评估 DESIGN-v5.md 是否可开始编码

### 评估结论
DESIGN-v5 整体架构清晰，方向正确（废弃独立 Orchestrator、Agent 自编排、task 原语），MVP 范围合理（3-4 周）。**可以开始写代码，但需先明确 3 个关键问题**。

### 发现的 7 个问题（按阻塞程度排序）

1. **父子 Agent 成本追踪不联通** — `spawnSubAgent` 传入了 `costTracker.remaining` 作为预算，但新 `AgentLoop` 会创建新的 `CostTracker`，子 Agent 花费不会扣减父 Agent 预算。建议 `CostTracker` 改为引用类型或让子 Agent 花费回传累加。

2. **`task` 工具 schema 未定义** — AgentLoop 处理了 `tc.name === "task"`，但模型需要 task 工具的 JSON schema 才会调用它。需在文档中补充 schema 定义。

3. **全局权限与 Agent 级权限优先级不明** — `orchestrator.yaml` 的 `requireApproval` 和 Agent 定义中的 `tools` 权限冲突时听谁的？建议：全局配置是最低安全基线，Agent 配置只能更严格。

4. **`AdapterSelector.select` 逻辑冗余** — `AgentDefinition.provider` 不是可选字段，`if (definition.provider)` 永远为 true。

5. **流式响应在 AgentLoop 中缺失** — `chatStream` 定义了但 `AgentLoop.run()` 未使用，MVP 是否真正支持流式需明确。

6. **`maxConcurrentAgents` 未在代码中体现** — 配置定义了但 `spawnSubAgent` 中无并发控制。

7. **单次模型调用可能突破预算** — `costTracker.record()` 在调用后执行，单次大输出可能超预算。可接受但需文档说明。

### 建议
先明确问题 1-3，然后立即开写。其余 4 个可在编码中顺手修。
